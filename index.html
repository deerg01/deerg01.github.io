Challenge 6 – Scenergy (macOS) *In progress

App Store ↗
Stack
Algorithms AIML PDF Parsing Classification macOS PDFKit Foundation Model Swift Swift Data
Overview

In film production, converting scenario(text) into structured scene lists (tabular data) has been mostly manual. Scenergy aims to automate that process — letting users upload a scenario PDF and receive a draft scene-list matching a common Excel format through keyword classification and context inference.

Key Challenges
  • Parsing diverse scenario formats: handling inconsistent headers and varying textual structures (directions, dialogue, narration) found in real scripts.
  • Scene-label extraction: designing an algorithm to detect locations, characters, and temporal cues from NL text before applying any AI model.
  • ML exploration & iteration: improving metadata extraction through CreateML models while comparing hybrid, rule-based, and FM-based inference strategies.
Approach
  • Modular parser: PDFKit text extraction with structural heuristics; Vision-based OCR fallback for problematic fonts and embedded scans.
  • Token classification pipeline: regex/FSM-based detectors refined with context rules and CreateML classifiers for higher labeling consistency.
  • AI inference layer: iterative experiments comparing CreateML and Foundation Model for keyword classification and implicit metadata inference.
  • macOS UX optimization: shortcut-driven navigation, reduced input states, and processing loops optimized for minimal user effort.
  • Data structure for minimal user input: designed representations for scenario sources, editable scene-list structures, and export-ready CSV formats; implemented project-level JSON schemas and SwiftData storage.
Outcome (In progress)
  • Current status: parser and labeling pipeline fully functional; CreateML + FM inference stack stable; UX tests show significantly reduced manual editing time for script-to-scene-list workflows.
  • Implemented a robust PDFKit based parser that filters vector noise, extracts clean text, and segments content into Foundation Model–compatible chunks without losing structural meaning.
  • Built a rule-based + ML hybrid labeling system that achieved FM-comparable accuracy when matched against a curated ground-truth set. Despite the strong performance, Foundation Models were selected for production due to their superior ability to generate summaries and implicit metadata.
  • Implemented a scenario → scene-list data flow optimized for minimal user input, including JSON-based project structures, editable scene entities, CSV import/export routines, and SwiftData integration for persistence and fast updates.

Challenge 4 – Rootrip (iOS)

Git Repository ↗ YouTube ↗
Stack
Swift SwiftUI MapKit PencilKit Firebase/Firestore SwiftData Git
Overview

Rootrip unifies trip planning in one app — map exploration, route lookup, itinerary creation, and real-time collaboration. It began as an effort to define missing capabilities, extend Apple frameworks, and build a coherent system, rather than simply assembling features.

Key Challenges
  • MapKit overlay limits: no native per-segment styling or live updates on edited paths.
  • PencilKit bitmap output: freehand strokes lacked closed-area recognition and vector scalability.
  • Data consistency: multi-user planning required local (SwiftData) ↔ cloud (Firestore) sync with conflict policy.
Approach
  • Repository pattern: decoupled view models from storage; local-first PKStroke in SwiftData, commit at PKDrawing; itineraries sync instantly to Firestore.
  • Custom path rendering: subclassed MKOverlayRenderer and overrode draw (CoreGraphics) for per-segment color/width with zoom-safe AA/scale.
  • Closed-area detection: vectorized PKStrokePoint, ray-casting polygon close, area thresholding, and interpolated boundaries on zoom.
  • Overlay orchestration: OverlayManager to prioritize annotations/drawings/routes with show/hide and state-based updates.
Outcome
  • Delivered a multi-module, maintainable codebase and a responsive, layered map UI.
  • Evolved from app assembly to framework extension, adding the pieces the SDK lacked.

Graduation Project – Mixed Precision BERT Quantization

Stack
Transformer Model BERT NLP Quantization QAT / PTQ PyTorch GLUE / MRPC
Overview

This project investigates mixed-precision quantization-aware training (QAT) for transformer models, enabling BERT to dynamically choose between INT4 and INT8 depending on layer sensitivity. Using BERT-base-uncased on the MRPC (GLUE) dataset, the goal was to reduce computation cost while preserving accuracy across FP32, INT8, INT4, and mixed-precision schemes.

Key Challenges
  • Transformer sensitivity to low precision: identifying which layers can tolerate INT4 and which require INT8.
  • Stable QAT integration: injecting fake quantization modules without breaking transformer forward/backward passes.
  • Adaptive precision rules: designing logic that lets the model select INT4/INT8 from weight and activation characteristics.
  • Accuracy preservation: minimizing loss spikes and accuracy degradation during mixed-precision training.
Approach
  • Architectural grounding: reviewed core transformer and LLM papers to deepen understanding of self-attention, activation flow, and layer sensitivity, forming hypotheses for where low precision is safe.
  • Per-layer analysis: examined before/after activation ranges, weight distributions, and input token sizes for each layer to measure tolerance to precision reduction in terms of stability and accuracy.
  • QAT integration: implemented CustomQuantizationLayers across embeddings, self-attention, and FFN modules, applying fake quantization and clipping strategies during training.
  • Adaptive precision selection: implemented multiple rules — weight-based (INT4/INT8 from weight spread and sensitivity), sampling-based (only a ratio of submodules in INT4), and a predefined baseline — then compared their runtime and accuracy impact.
Outcome
  • Built a Mixed-precision BERT model that automatically selects INT4 or INT8 per layer, reducing computation while keeping accuracy close to FP32.
  • confirmed that self-attention layers generally require higher precision, while many FFN layers operate safely in INT4.
  • showed that adaptive mixed precision clearly outperforms uniform INT4 in stability and overall performance.
